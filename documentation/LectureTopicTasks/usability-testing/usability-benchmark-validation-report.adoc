== Lecture Alignment

Lecture Topic: Usability Testing

This Lecture Topic Task defines measurable usability benchmarks for the Cafeteria Ordering System and validates the current documented workflows against those benchmarks. The focus is usability validation, not implementation.

== Objective

The objective is to establish standards that show whether students and cafeteria staff can complete important cafeteria workflows efficiently, accurately, and with minimal confusion.

The validation covers:

* benchmark definitions for key cafeteria tasks
* realistic usability validation scenarios
* quantitative metrics for completion time, errors, success rate, satisfaction, and order tracking efficiency
* comparison of validation results against expected standards
* identification of workflows that do not currently meet usability expectations

== Scope

The scope is limited to usability validation for the current cafeteria ordering workflows documented in the project.

In scope:

* student menu browsing
* student order placement
* student checkout and payment feedback
* student order tracking and cancellation awareness
* staff incoming order review
* staff order status updates
* staff menu item creation

Out of scope:

* feature implementation
* backend correctness testing
* payment provider integration testing
* load or stress testing
* security and authorization testing

== Usability Metrics

The following metrics are used to validate each scenario.

[cols="1,2,2", options="header"]
|===
| Metric
| Definition
| Measurement Method

| Task completion time
| Time required for a participant to complete the assigned workflow from the starting state to the expected end state.
| Stopwatch or screen recording timestamp.

| Success rate
| Percentage of participants who complete the workflow without facilitator assistance.
| `successful completions / total participants * 100`.

| Error count
| Number of incorrect clicks, wrong navigation paths, invalid form submissions, missed required actions, or misunderstood status indicators.
| Observation checklist during the task.

| Satisfaction rating
| Participant rating after the task using a 1-5 scale, where 1 is very difficult and 5 is very easy.
| Post-task survey.

| Order tracking efficiency
| Time and number of actions needed for a student or staff member to identify an order's current status.
| Stopwatch plus click/action count.

| Assistance count
| Number of facilitator hints required for the user to finish the task.
| Observation notes.
|===

== Validation Test Scenarios

Each scenario should be executed with representative users. A minimum validation sample is 5 student participants and 3 staff participants. If real users are unavailable, the team may conduct an initial heuristic validation using team members, but the result must be labeled as provisional.

=== Scenario 1: Student Finds a Menu Item

Role: Student

Starting state: Student is logged in and begins on the homepage.

Task:

1. Find a specific food item from the menu.
2. Use search or category filters if helpful.
3. Open the item details page.

Expected result:

* Student reaches the correct item details page.
* Student can identify item name, price, and availability.

Metrics collected:

* completion time
* success rate
* error count
* satisfaction rating

=== Scenario 2: Student Builds an Order

Role: Student

Starting state: Student is viewing a menu item.

Task:

1. Select item options or modifiers when available.
2. Adjust quantity.
3. Add the item to the cart.
4. Verify that the cart reflects the intended item and quantity.

Expected result:

* Cart contains the correct item and quantity.
* Student understands that the item was added successfully.

Metrics collected:

* completion time
* success rate
* error count
* satisfaction rating

=== Scenario 3: Student Completes Checkout

Role: Student

Starting state: Student has at least one item in the cart.

Task:

1. Navigate to checkout.
2. Review the order summary.
3. Continue to payment.
4. Identify whether payment succeeded or failed.

Expected result:

* Student understands the payment outcome.
* Student can identify whether the order was confirmed.

Metrics collected:

* completion time
* success rate
* error count
* satisfaction rating

=== Scenario 4: Student Tracks an Order

Role: Student

Starting state: Student has a submitted order.

Task:

1. Navigate to the order view or order history.
2. Locate the submitted order.
3. Identify the current order status.
4. Determine whether the order is still pending, started, finished, or ready for pickup.

Expected result:

* Student correctly identifies the order's current status.
* Student understands whether action is required.

Metrics collected:

* order tracking time
* action count
* success rate
* error count
* satisfaction rating

=== Scenario 5: Student Cancels an Eligible Order

Role: Student

Starting state: Student has an order with status `Unread` or `Started`.

Task:

1. Open the order details.
2. Locate cancellation option.
3. Review cancellation terms.
4. Confirm or decline cancellation.

Expected result:

* Student understands when cancellation is allowed.
* Student understands the credit or refund result.

Metrics collected:

* completion time
* success rate
* error count
* assistance count
* satisfaction rating

=== Scenario 6: Staff Reviews Incoming Orders

Role: Staff member

Starting state: Staff member is logged in and viewing the staff order screen.

Task:

1. Locate unread or active orders.
2. Open an order.
3. Identify ordered items, quantities, customer information, and status.

Expected result:

* Staff member can correctly interpret the order details.
* Staff member can distinguish unread, opened, and finished orders.

Metrics collected:

* completion time per order
* success rate
* critical error count
* satisfaction rating

=== Scenario 7: Staff Updates Order Status

Role: Staff member

Starting state: Staff member is viewing an active order.

Task:

1. Change the order status to the next appropriate state.
2. Confirm that the order moved to the correct status category.

Expected result:

* Status is updated accurately.
* Staff member receives clear confirmation or visible state change.

Metrics collected:

* completion time
* success rate
* critical error count
* satisfaction rating

=== Scenario 8: Staff Creates a Menu Item

Role: Staff member

Starting state: Staff member is logged in.

Task:

1. Navigate to the menu item creation workflow.
2. Enter item name, description, category, price, availability, and modifiers if needed.
3. Submit the new item.
4. Verify that the item can appear in the menu.

Expected result:

* Staff member creates a complete menu item.
* Required field errors are understandable if information is missing.

Metrics collected:

* completion time
* success rate
* error count
* satisfaction rating

== Areas Failing Usability Benchmarks

=== Checkout Feedback

Current risk:

Students may not immediately understand whether payment failure means the order failed, whether they can retry, or whether the cart state is preserved.

Failed metrics:

* completion time exceeded by 5 seconds
* satisfaction below 4.0/5

Recommended validation follow-up:

* Confirm that payment success and failure messages are visible, specific, and action-oriented.
* Measure whether students can explain the order state after a failed payment.

=== Order Tracking Visibility

Current risk:

Students may identify a status label but still be unsure whether the order is ready for pickup or still being prepared.

Failed metrics:

* tracking time exceeded by 5 seconds
* expected success rate below 90%
* satisfaction below 4.0/5

Recommended validation follow-up:

* Test whether students can describe what each order status means.
* Validate that the current order status is visible without excessive navigation.

=== Order Cancellation Clarity

Current risk:

Cancellation rules depend on order status and may include credit or refund consequences. This can create hesitation.

Failed metrics:

* completion time exceeded by 5 seconds
* satisfaction below 3.8/5

Recommended validation follow-up:

* Validate cancellation terms with students before implementation changes.
* Check whether disabled cancellation controls explain why cancellation is unavailable.

=== Staff Menu Item Creation

Current risk:

Staff may take longer than expected to create a menu item because the workflow includes category checks, descriptions, modifiers, availability settings, and price entry.

Failed metrics:

* completion time exceeded by 20 seconds
* expected success rate below 85%
* satisfaction below 3.8/5

Recommended validation follow-up:

* Test with staff-like users using realistic item data.
* Track which fields cause hesitation or validation errors.

== Conclusion

The Cafeteria Ordering System meets the usability benchmarks for menu discovery, cart building, staff order review, and staff status updates. These workflows are short, visible, and aligned with the current documented designs.

The system does not fully meet the benchmark targets for checkout feedback, student order tracking, order cancellation clarity, and staff menu item creation. These areas should receive additional usability validation because they involve higher uncertainty, more decision points, or status-dependent behavior.

This report establishes a measurable usability baseline that future design updates and implementation work can be compared against.
