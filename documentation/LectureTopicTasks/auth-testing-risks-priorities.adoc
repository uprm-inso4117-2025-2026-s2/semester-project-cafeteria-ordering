== Objective

Identify authentication-related risks and prioritize testing efforts using a risk-based approach.

Risk Score is calculated as:

    Risk Score = Likelihood (1–5) × Impact (1–5)

Higher scores indicate higher testing priority and broader regression scope.

== Risk Justification

=== R1: Unauthorized Login Access
Authentication involves multiple flows (login, guest access, RBAC), increasing complexity and probability of defects. Impact is critical due to exposure of user data and order manipulation.

=== R2: Weak Password Validation
Common implementation gap. High likelihood if validation rules are not enforced. High impact due to account compromise risk.

=== R3: Session Expiration Issues
Session handling is often overlooked. High impact in shared/public device scenarios.

=== R4: Brute Force Vulnerability
Very likely without rate limiting. Can lead to account compromise and system strain.

=== R5: RBAC Failure
System includes multiple roles (Customer vs Staff). Misconfiguration can expose administrative functionality.

=== R6: Guest Access Issues
Guest flow adds complexity to identity handling. Medium impact due to inconsistencies in order tracking and verification.

== Testing Priority Strategy

Based on risk scores:

- *High (13–25):* Must be tested first. Include in smoke + regression suites.
- *Medium (6–12):* Test after high-priority risks. Include in targeted regression.
- *Low (1–5):* Minimal testing unless impacted by changes.

== Relation to Regression Testing

High-risk authentication areas must be included in regression testing when:

- Authentication logic changes
- Role/permission logic is modified
- Session management is updated

Examples of regression coverage:

- Login/logout flows
- Role-based access restrictions
- Guest vs authenticated user behavior
