= TDD Specification for MenuItem Class Data Retrieval

== 1. Overview of TDD

Test-Driven Development (TDD) is a software development practice in which tests are written before the implementation.
This document uses TDD with pseudocode examples from the `MenuItem` class to ensure that it is accurate.

== 2. MenuItem Class Context

=== 2.1 Selected Unit

[source]
----
fromDatabaseRow : MenuItemRow >< IngredientItem[] → MenuItem
----

This operation corresponds to:

* *Action:* Construction of a `MenuItem` object from a Supabase row and its joined ingredients
* *Events:* Valid row mapped successfully / Unavailable item correctly rejected
* *Behavior:* `MenuItem.fromDatabaseRow` receives a raw `MenuItemRow` from the `menu_items`
  table and an `IngredientItem[]` joined through `menu_items_ingredients_relationship`,
  and returns a fully constructed `MenuItem` instance

=== 2.2 Inputs and Outputs

* *Inputs:* `MenuItemRow` (raw Supabase row), `IngredientItem[]` (joined ingredient records)
* *Outputs:* A fully constructed `MenuItem` instance with all fields populated and all getters functional

=== 2.3 Initial Informal Specification

* If a valid `MenuItemRow` and ingredient list are provided, a `MenuItem` must be returned with all fields correctly mapped.
* The returned item's `id`, `name`, and `price` must exactly match the values in the source row.
* The returned item's `available` flag must be correctly reflected by `isAvailable()` and `canOrder()`.

== 3. TDD Test Cases — Incremental Construction

=== Test 1 — A Valid Row Produces a MenuItem Instance

*Test:* Given a valid `MenuItemRow` and `IngredientItem[]`, calling
`MenuItem.fromDatabaseRow(row, ingredients)` should return a `MenuItem` instance.

*Dummy Implementation:*
[source, typescript]
----
static fromDatabaseRow(row: MenuItemRow, ingredients: IngredientItem[]): MenuItem {
  return new MenuItem(); // blank constructor — returns empty item
}
----

*Result:* Test fails — the returned object exists but no fields are mapped from the row.

*Minimal Implementation:*
[source, typescript]
----
static fromDatabaseRow(row: MenuItemRow, ingredients: IngredientItem[]): MenuItem {
  return new MenuItem(
    row.id, row.category_id, row.name, ingredients, row.price,
    row.image_url, row.available, row.allergens, row.prep_time_minutes,
    row.created_at, row.updated_at
  );
}
----

Test 1 now passes.

---

=== Test 2 — The Returned Item Must Correctly Map id, name, and price

The first test only confirmed that an instance is returned. Now we check that data is actually mapped correctly from the source row.

*Test:* The values returned by `getId()`, `getName()`, and `getBasePrice()` must exactly
match `row.id`, `row.name`, and `row.price` respectively.

[source, typescript]
----
const item = MenuItem.fromDatabaseRow(row, []);
assert(item.getId() === "1e7578f3-ad2c-4085-8c23-ef600e8d4c93");
assert(item.getName() === "Chicken Wrap");
assert(item.getBasePrice() === 10);
----

*Result:* Tests pass — field mapping is correctly handled by the full constructor.

Tests 1–2 now pass.

---

=== Test 3 — An Unavailable Item Is Correctly Reflected by isAvailable() and canOrder()

The business rule that unavailable items must not be orderable by customers is now enforced.

*Test:* Given a `MenuItemRow` where `available = false`, the constructed `MenuItem` must
return `false` from both `isAvailable()` and `canOrder()`.

[source, typescript]
----
const unavailableRow: MenuItemRow = { ...row, available: false };
const item = MenuItem.fromDatabaseRow(unavailableRow, []);
assert(item.isAvailable() === false);
assert(item.canOrder() === false);
----

*Result:* Test passes — `available` is mapped directly from the row and `canOrder()`
checks `this.available` internally:

[source, typescript]
----
canOrder(): boolean {
  if (!this.available) {
    console.warn(`"${this.name}" is currently unavailable.`);
    return false;
  }
  return true;
}
----

Tests 1–3 now pass.

== 4. Refactoring Analysis

With behavior stabilized by tests, the inline existence check inside `addIngredient` is
extracted into a named private helper to eliminate duplication.

*Before:*
[source, typescript]
----
addIngredient(ingredient: IngredientItem): void {
  const exists = this.ingredients.some(
    (ing) => ing.ingredients_id === ingredient.ingredients_id
  );
  if (exists) {
    console.warn(`Ingredient "${ingredient.ingredients_names}" is already in "${this.name}".`);
    return;
  }
  this.ingredients.push(ingredient);
  if (ingredient.ingredients_price) {
    this.price += ingredient.ingredients_price;
  }
}
----

*After — extract hasIngredient():*
[source, typescript]
----
private hasIngredient(ingredientId: string): boolean {
  return this.ingredients.some((ing) => ing.ingredients_id === ingredientId);
}

addIngredient(ingredient: IngredientItem): void {
  if (this.hasIngredient(ingredient.ingredients_id)) {
    console.warn(`Ingredient "${ingredient.ingredients_names}" is already in "${this.name}".`);
    return;
  }
  this.ingredients.push(ingredient);
  if (ingredient.ingredients_price) {
    this.price += ingredient.ingredients_price;
  }
}
----

== 5. Result

Through three incremental tests:

* `fromDatabaseRow` evolved from a blank stub into a fully verified construction pipeline.
* Each change was driven by a failing test.
* The final unit now reflects the domain rules of the Cafeteria Ordering System:
** Valid rows are fully mapped with all fields accessible through typed getters.
** Unavailable items are correctly rejected at the `canOrder()` boundary.
** Internal duplication was safely eliminated through refactoring without breaking any test.
