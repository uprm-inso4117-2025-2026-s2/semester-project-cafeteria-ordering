= Payment Screen Load and Performance Testing Report

== Introduction

This report presents the results of testing conducted on the payment flow of the Cafeteria Ordering System, based on the design of the payment screen. The goal was to understand how the system behaves under different levels of user activity and whether it remains responsive and reliable during both normal and peak usage.

To do this, three types of tests were performed: performance testing with a single user, load testing with a realistic number of users, and stress testing with higher-than-expected usage.

== Testing Approach

The testing process was designed to simulate realistic user behavior when submitting a payment. Since the payment screen is not yet connected to a real backend and full UI-based testing was not configured, a mock payment service and request-based scripts were used to replicate the expected behavior of the payment process.

These scripts simulate how the payment screen would send data to a backend, using the same structure and inputs defined in the payment screen implementation.

Three different scenarios were tested:
- A single-user scenario to measure baseline performance
- A moderate multi-user scenario to represent typical usage
- A high-load scenario to push the system beyond normal limits

For each scenario, the same key metrics were collected so the results could be compared.

== Metrics Collected

The following metrics were used to evaluate performance:

- *Response Time*: how long it takes for a payment request to be processed
- *Latency*: the delay between sending a request and receiving a response
- *Throughput*: how many payments are processed per second
- *Failure Rate*: the percentage of requests that fail

These metrics help give a clear picture of both speed and reliability.

== Analysis

Overall, the system performs well under normal and expected usage conditions. The performance test shows that response times are consistent when only one user is active, which suggests the system is stable at a basic level.

During the load test, the system scaled effectively, handling multiple users without a major impact on response time. The small increase in failures is noticeable but still within a reasonable range.

The stress test shows what happens when the system is pushed beyond its intended capacity. While it continues to process requests at a high rate, the increase in failures indicates that the system begins to struggle under extreme conditions. This is expected behavior and helps identify where improvements may be needed.

== Conclusion

The payment flow demonstrates strong performance under normal and moderately high usage. It is able to handle expected levels of traffic while maintaining fast response times and low failure rates.

Under extreme conditions, the system continues to function but shows signs of reduced reliability. This suggests that while the current implementation is sufficient for typical use, further improvements could be made to better handle peak demand.

Future work includes validating these results against the fully integrated application once backend connectivity and UI-level testing are fully implemented.
