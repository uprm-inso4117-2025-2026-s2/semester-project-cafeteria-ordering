= Payment Screen Load and Performance Testing Report

== Introduction

This report presents the results of testing conducted on the payment flow of the Cafeteria Ordering System, based on the design of the payment screen. The goal was to understand how the system behaves under different levels of user activity and whether it remains responsive and reliable during normal and peak usage.

To do this, three types of tests were performed: performance testing with a single user, load testing with a realistic number of users, and stress testing with higher-than-expected usage.

== Testing Approach

The testing process was designed to simulate realistic user behavior when submitting a payment. Since the payment screen is not yet connected to a real backend and full UI-based testing was not configured, a mock payment service and request-based scripts were used to replicate the expected behavior of the payment process.

These scripts simulate how the payment screen would send data to a backend, using the same structure and inputs defined in the payment screen implementation.

Three different scenarios were tested:
- A single-user scenario to measure baseline performance
- A moderate multi-user scenario to represent typical usage
- A high-load scenario to push the system beyond normal limits

For each scenario, the same key metrics were collected so the results could be compared.

== Metrics Collected

The following metrics were used to evaluate performance:

- *Response Time*: how long it takes for a payment request to be processed
- *Latency*: the delay between sending a request and receiving a response
- *Throughput*: how many payments are processed per second
- *Failure Rate*: the percentage of requests that fail

These metrics help give a clear picture of both speed and reliability.

== Analysis

Overall, the results show that the payment flow remained stable across all three testing scenarios. In the performance test, response times were consistent and there were no failures, which suggests that the system behaves well under basic conditions.

During the load test, the system handled a much larger number of requests while still keeping response time and latency near the same range as the performance test. Although the failure rate increased to 2.67%, the success rate remained at 97.33%, which suggests that the system can still handle expected usage reasonably well.

The stress test produced the highest throughput. Even so, the average response time stayed close to the other scenarios and the success rate remained high at 97.94%. This suggests that the mock payment flow scaled well during testing, although the presence of some failures still shows that reliability can begin to drop as load increases.

One notable observation is that the failure rate in the stress test is slightly lower than in the load test, even though the system is under heavier load. This is likely due to the randomness of the mock simulation rather than an indication that the system performs better under stress.

== Conclusion

The payment flow demonstrates strong performance under normal, expected, and high-usage conditions. It is able to handle increasing levels of traffic while maintaining relatively fast response times and high success rates.

Although some failures appeared during the load and stress tests, the overall results suggest that the current implementation is capable of supporting typical cafeteria usage and can remain responsive even under heavier demand.

Future work includes validating these results against the fully integrated application once backend connectivity and UI-level testing are fully implemented.
