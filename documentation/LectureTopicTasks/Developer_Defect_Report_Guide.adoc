= Define and establish Developer Defect Reports

*By: Gerardo Y. Soto Ríos*

== Introduction
Establishing a proper developer defect report allows for a quicker resolution of issues that may affect the project in the long term. By having a detailed and professional document that captures unintended outputs found during development or inspection, we ensure that the responsible team is well informed and able to resolve issues in an efficient manner.

Defect priority and severity must be correctly identified using the document:
`documentation/QualityAssurance/Test_Plan/test_strategy/testing_defects.adoc`

== Defect Report Structure

[cols="2,3", options="header"]
|===
| Field | Description
| Short Name | Brief descriptive title of defect
| Date Found | MM/DD/YYYY
| Team Responsible | Name of the team
| Priority | High/Medium/Low
| Severity | Blocker/Critical/Major/Minor/Trivial
| Issue Related | number or link
| Description | Detailed explanation with example
| Steps to Reproduce | Step-by-step guide
|===

== Conclusion
By using this developer defect report structure, teams ensure that defect issues are clearly documented and efficiently communicated. This approach supports consistent tracking and faster resolution of defects throughout the development process. Ultimately, it helps maintain project stability and ensures progress aligns with defined milestone goals.
