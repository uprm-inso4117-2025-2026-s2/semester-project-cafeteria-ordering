= Define Risk-Based Test Priorities for Staff View

== Purpose
Apply risk-based testing to the Staff View of the Cafeteria Ordering System. Focus on staff-facing functions whose failure would most disrupt operations, create inconsistent data, or reduce trust. For each function, record Likelihood and Impact, calculate a Risk Score (`L x I`), and define the resulting test focus.

This follows the lecture idea that testing resources are limited and tests should be prioritized according to risk, with the goal of finding high-cost defects first. :contentReference[oaicite:0]{index=0}

== Scoring Rubric
*Likelihood (L) 1-5:* Probability of a defect occurring (`1 = Rare` … `5 = Very Likely`)  
*Impact (I) 1-5:* Severity if the function fails (`1 = Negligible` … `5 = Critical`)  
*Risk Score:* `L x I` (`1-25`); higher means higher priority.

== Testing Priority Groups
*High Priority:* Closing orders, Viewing orders, Editing menu items  
*Medium Priority:* Adding menu items, Viewing menu items  
*Low Priority:* None in scope

== Example Failure Consequences
* Staff unable to close orders  
* Incorrect menu updates  
* Missing or outdated item information  
* Data inconsistencies between views  

== How Priorities Guide Testing Order
High-risk functions should be tested first across levels identified in the lecture: unit, integration, and system testing. :contentReference[oaicite:1]{index=1}

* Unit testing should start with `Closing orders` and `Editing menu items`, focusing on transition logic, validation, and update behavior.
* Integration testing should next focus on `Closing orders`, `Viewing orders`, and `Editing menu items`, especially where UI, backend, and persistence interact.
* System testing should execute end-to-end scenarios in this order:
. Closing orders
. Viewing orders
. Editing menu items
. Adding menu items
. Viewing menu items
