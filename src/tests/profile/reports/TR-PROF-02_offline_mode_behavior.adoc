== Validated Implemented Behavior
The current implementation supports the following validated offline-related behavior on the profile screen:

- Profile information is loaded from `@profile_info` in AsyncStorage on mount
- Avatar information is loaded from `@profile_avatar` in AsyncStorage on mount
- Missing cached profile data falls back safely to default profile UI state
- Missing cached avatar data falls back safely to the default avatar/initial state
- Previously cached profile data can be recovered on a fresh mount/remount
- Previously cached avatar data can be recovered on a fresh mount/remount
- Pressing `Edit` on the current profile screen navigates to `/edit-profile`
- The current profile screen does not perform inline local save of profile edits
- Permission denial during avatar selection shows the expected alert and prevents avatar selection from proceeding

== Unsupported / Not Implemented Behavior
The following offline-related behaviors are not currently implemented on the profile screen and were therefore not validated as supported behavior:

- Reconnect-based backend synchronization for profile edits
- Retry queues for pending offline profile changes
- Conflict handling between local profile edits and backend profile state
- Duplicate-prevention logic for replayed offline edits
- Offline/online status banner or explicit offline messaging in the profile screen
- Automatic backend update of profile edits from the current profile screen
- Inline edit-and-save workflow on `src/app/(tabs)/profile.tsx`

== Conclusion
The current implementation satisfies a limited but valid offline profile behavior scope through AsyncStorage-based recovery of cached profile and avatar data, safe fallback behavior when cached data is missing, and correct navigation from the profile screen to the dedicated `/edit-profile` flow.

This means the current profile screen supports:

- offline access to previously cached profile data
- offline access to previously cached avatar state
- safe default rendering when cache entries are absent
- navigation to the dedicated edit flow rather than inline editing on the profile screen

However, the broader offline synchronization behavior described in the issue is only partially supported by the present implementation. Full reconnect-based profile synchronization, retry handling, and conflict handling are not currently available in the profile screen implementation and therefore were not validated as implemented behavior.
